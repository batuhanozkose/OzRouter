# Research: Electron OTA Auto-Update System

## Summary

`electron-updater` (electron-builder ecosystem) is the standard solution for OTA auto-updates in Electron apps. It supports GitHub Releases, S3, generic HTTP servers; works on Windows (NSIS), macOS (DMG+ZIP), and Linux (AppImage/DEB/RPM). OzRouter already has a working basic integration â€” `electron-updater@6.8.3` installed, `autoUpdater` configured in `electron/main.js` with IPC events, and GitHub publish config in `electron/package.json`. Key gaps: no channel/staging strategy, no code signing, no differential update optimization, missing `zip` target for macOS.

## Findings

### 1. electron-updater â€” How It Works

1. **Architecture** â€” `electron-updater` is a separate package from Electron's built-in `autoUpdater`. It reads `app-update.yml` (auto-generated by electron-builder in `resources/` during build) to know where to check for updates. No `setFeedURL()` needed. [electron-builder docs](https://www.electron.build/auto-update)

2. **Platform-specific updaters** â€” Internally dispatches to:
   - `NsisUpdater` (Windows)
   - `MacUpdater` (macOS)
   - `AppImageUpdater`, `DebUpdater`, `RpmUpdater`, `PacmanUpdater` (Linux)
     [electron-updater API](https://mintlify.com/electron-userland/electron-builder/api/electron-updater)

3. **Advantages over built-in autoUpdater**:
   - Linux support (built-in has none)
   - Code signature validation on Windows too (not just macOS)
   - Download progress + staged rollouts on all platforms
   - Multiple providers: GitHub, S3, DigitalOcean Spaces, Keygen, generic HTTP(s)
   - 2 lines of code to work
     [electron-builder docs](https://www.electron.build/auto-update)

4. **Key configuration options**:

   ```typescript
   autoUpdater.autoDownload = false; // Let user choose
   autoUpdater.autoInstallOnAppQuit = true; // Install on quit
   autoUpdater.allowDowngrade = false; // Prevent downgrade
   autoUpdater.allowPrerelease = false; // Pre-release channels
   autoUpdater.channel = "latest"; // Update channel
   autoUpdater.forceDevUpdateConfig = true; // Use dev-app-update.yml
   ```

5. **Best practices**:
   - Delay update check 3-5s after app launch (avoid lock issues on Windows)
   - Check periodically (every 1-4 hours)
   - Never check on first install (`--squirrel-firstrun` flag on Windows)
   - Use `autoDownload = false` for user control
   - Communicate progress via IPC to renderer
   - Log with `autoUpdater.logger = electronLog` for debugging
     [emadibrahim.com guide](https://emadibrahim.com/electron-guide/auto-updates)

### 2. GitHub Releases Setup

6. **Configuration** â€” Add `publish` to electron-builder config:

   ```json
   {
     "build": {
       "publish": {
         "provider": "github",
         "owner": "batuhanozkose",
         "repo": "OzRouter"
       }
     }
   }
   ```

   OzRouter already has this âś…. [electron-builder publish docs](https://www.electron.build/publish)

7. **Token setup** â€” Set `GH_TOKEN` or `GITHUB_TOKEN` env var. For CI, use repository secrets. For fine-grained tokens: "Contents" permission with read+write is sufficient. Optional `GITHUB_RELEASE_TOKEN` for separation of publish vs update-check tokens. [electron-builder publish docs](https://www.electron.build/publish)

8. **Build & publish commands**:

   ```bash
   # Build + publish to GitHub Releases
   electron-builder --publish always

   # Build + publish on git tag only
   electron-builder --publish onTag

   # Build without publishing
   electron-builder --publish never
   ```

   âš ď¸Ź Implicit publish behavior is deprecated in v27. Always use `--publish` flag explicitly. [electron-builder docs](https://www.electron.build/publish)

9. **Private repos** â€” Set `GH_TOKEN` env var on user machine + `private: true` in publish config. Uses GitHub API for updates (rate limit: 5000 req/user/hr, ~3 req per check). Not suitable for large user bases. [electron-builder docs](https://www.electron.build/auto-update.html)

10. **Draft releases** â€” Publish as draft first, verify artifacts, then make public. Set `releaseType: "draft"` or env `EP_DRAFT=true`. [electron-builder docs](https://electron.build/builder-util-runtime.Interface.GithubOptions.html)

### 3. electron-builder Integration

11. **Metadata files** â€” Build produces `latest.yml` (Windows), `latest-mac.yml` (macOS), `latest-linux.yml` (Linux) alongside installers. These are the update manifests. [electron-builder docs](https://www.electron.build/auto-update)

12. **OzRouter current config analysis**:
    - âś… `electron-updater@6.8.3` installed as dependency
    - âś… GitHub publish config present
    - âś… NSIS target for Windows (auto-update compatible)
    - âś… AppImage + DEB for Linux (auto-update compatible)
    - âš ď¸Ź macOS uses `dmg` only â€” **needs `zip` target added** for `latest-mac.yml` generation
    - âš ď¸Ź No `--publish` flag in build scripts

13. **macOS fix needed** â€” Current config:

    ```json
    "mac": { "target": ["dmg"] }
    ```

    Should be:

    ```json
    "mac": { "target": ["dmg", "zip"] }
    ```

    Without `zip`, `latest-mac.yml` won't be created â†’ `autoUpdater` errors on macOS. Default is `dmg+zip`, OzRouter explicitly overrode to `dmg` only. [electron-builder docs](https://www.electron.build/auto-update)

14. **Dev testing** â€” Create `dev-app-update.yml` in project root matching publish config (YAML format). Set `autoUpdater.forceDevUpdateConfig = true` to test without packaging. [electron-builder docs](https://www.electron.build/auto-update)

### 4. Differential/Delta Updates

15. **Blockmap-based differential downloads** â€” electron-builder generates `.blockmap` files alongside installers. When updating, electron-updater downloads only changed blocks instead of full installer. Works automatically for NSIS (Windows) and DMG/ZIP (macOS). [electron-builder issue #2851](https://github.com/electron-userland/electron-builder/issues/2851)

16. **How blockmaps work** â€” Gzipped JSON containing base64-encoded hashes + block sizes. Old and new blockmaps compared â†’ only changed blocks downloaded. Typical savings: 60-90% bandwidth for minor updates. [electron-builder issue #2851](https://github.com/electron-userland/electron-builder/issues/2851)

17. **Known issues with blockmaps**:
    - Private GitHub repos: blockmap parsing fails (fixed in PR #6198)
    - "Cannot parse blockmap" errors common â€” fallback to full download
    - Can be disabled: `autoUpdater.disableDifferentialDownload = true`
    - `nsis-web` target supports `differentialPackage: true` for 7z delta
    - Regular NSIS uses blockmap approach only (not true delta)
      [electron-builder issues #6184, #4682](https://github.com/electron-userland/electron-builder/issues/6184)

18. **True delta updates** â€” `@electron-delta/builder` package uses binary diffing (HDiffPatch). Downloads only diff (~<1MB). Windows NSIS only. 90% bandwidth reduction vs blockmap's 60-70%. Not widely adopted, last release 2022. [electron-delta/electron-delta](https://github.com/electron-delta/electron-delta)

19. **Recommendation for OzRouter** â€” Keep blockmap approach (default, no extra config). If blockmap errors occur, differential falls back to full download automatically. Don't over-engineer delta updates for now.

### 5. Platform Differences

20. **Windows (NSIS)**:
    - Auto-update target: NSIS âś… (Squirrel.Windows NOT supported by electron-updater)
    - Code signing: Optional but strongly recommended (SmartScreen warnings without it)
    - Differential downloads: Blockmap-based, works well
    - First launch: `--squirrel-firstrun` lock â€” delay update checks 10s
    - Admin rights: Depends on `perMachine` setting
    - Portable target: NO auto-update support
      [Electron docs](https://electronjs.org/docs/latest/api/auto-updater)

21. **macOS**:
    - Auto-update target: DMG + ZIP (ZIP required for update manifest)
    - Code signing: **MANDATORY** â€” unsigned apps cannot auto-update (Squirrel.Mac requirement)
    - Notarization: Required for Gatekeeper
    - App Transport Security: Applies to update requests
    - Universal builds: Separate manifests per arch or universal
      [Electron docs](https://electronjs.org/docs/latest/tutorial/code-signing)

22. **Linux**:
    - Auto-update targets: AppImage âś…, DEB âś…, RPM âś…, Pacman (beta) âś…
    - Code signing: Not required
    - AppImage: Self-contained, update replaces file in-place
    - DEB/RPM: electron-updater handles, but package managers preferred for system packages
    - No built-in support in Electron's native `autoUpdater` â€” only via `electron-updater`
      [electron-builder docs](https://www.electron.build/auto-update)

### 6. Code Signing Requirements

23. **macOS code signing**:
    - **Required** for auto-update to work
    - Apple Developer Certificate ($99/year)
    - Needs signing + notarization
    - Env vars: `APPLE_ID`, `APPLE_ID_PASS`, `CSC_LINK`, `CSC_KEY_PASSWORD`
    - Tools: `@electron/osx-sign`, `@electron/notarize` (used by electron-builder)
      [Electron code signing docs](https://electronjs.org/docs/latest/tutorial/code-signing)

24. **Windows code signing**:
    - **Strongly recommended** (SmartScreen blocks unsigned apps)
    - Since June 2023: ALL code signing certs require HSM/hardware token (OV and EV both)
    - EV cert: ~$400-700/yr, instant SmartScreen trust
    - OV cert: ~$200-400/yr, builds reputation over time
    - **Azure Trusted Signing**: $9.99/mo, instant reputation, no hardware needed â€” **best option for 2026**
    - Cross-platform signing possible via `jsign` or `osslsigncode`
      [Electron code signing docs](https://electronjs.org/docs/latest/tutorial/code-signing), [Laststance.io guide](https://laststance.io/articles/How-to-Distribute-Electron-Apps-with-Code-Signing-on-Windows-and-Linux)

25. **Linux code signing** â€” Not required for auto-update. No OS-level enforcement.

26. **Publisher name migration** â€” Changing code signing cert publisher name breaks auto-update for existing users. Strategy: release intermediate version with both old+new publisher names, wait for adoption, then switch fully. [electron-builder issue #6499](https://github.com/electron-userland/electron-builder/issues/6499)

### 7. Staging/Canary Release Strategies

27. **Channel system** â€” electron-builder supports 3 built-in channels + custom:
    - `latest` (stable, default) â€” version: `1.3.2`
    - `beta` â€” version: `1.3.2-beta`
    - `alpha` â€” version: `1.3.2-alpha`
    - Custom: `dev`, `rc`, `stable` (limited support)

    Channel hierarchy: alpha users get alpha+beta+latest; beta users get beta+latest; latest users get latest only.
    [electron-builder channels docs](https://www.electron.build/tutorials/release-using-channels.html)

28. **Configuration for channels**:

    ```json
    // package.json
    {
      "version": "3.9.0-beta",
      "build": {
        "generateUpdatesFilesForAllChannels": true
      }
    }
    ```

    ```typescript
    // main.js â€” client side
    autoUpdater.channel = "beta"; // or from user settings
    autoUpdater.allowDowngrade = true; // allow switching from betaâ†’stable
    ```

    [electron-builder channels docs](https://www.electron.build/tutorials/release-using-channels.html)

29. **Staged rollouts** â€” Add `stagingPercentage: 10` to `latest.yml` manually. Probabilistic: user ID hashed â†’ bucket â†’ compared to percentage. Works well with large user bases. Small user base = unpredictable. No server required (serverless implementation). [electron-builder issue #3499](https://github.com/electron-userland/electron-builder/issues/3499)

30. **GitHub provider caveat** â€” `detectUpdateChannel` does NOT work with GitHub provider. Must explicitly set `publish.channel` or rely on `generateUpdatesFilesForAllChannels`. Known bug tracked in issue #8589. [electron-builder issue #8589](https://github.com/electron-userland/electron-builder/issues/8589)

31. **Recommended strategy for OzRouter**:
    - Use version-based channels: `3.9.0-beta.1`, `3.9.0-rc.1`, `3.9.0`
    - Set `generateUpdatesFilesForAllChannels: true`
    - Add in-app channel selector (Settings â†’ Update Channel)
    - Start with `latest` only, add beta channel when user base grows

### 8. Next.js + Electron Auto-Update

32. **OzRouter-specific architecture** â€” OzRouter bundles Next.js as standalone server inside Electron's `extraResources`. Update replaces the entire app including the bundled Next.js server. No separate web update mechanism needed â€” electron-updater handles everything. [OzRouter electron/package.json]

33. **Next.js as update server** â€” `@electron-releases/adapter-nextjs` package enables using Next.js App Router as a self-hosted update server. Creates Squirrel-compatible endpoints. Useful if OzRouter wanted to self-host updates instead of GitHub Releases:

    ```typescript
    // app/api/releases/[[...path]]/route.ts
    import { configureNextjsAdapter } from "@electron-releases/adapter-nextjs";
    import { configureGithubProvider } from "@electron-releases/provider-github";

    const provider = configureGithubProvider({
      token: process.env.GITHUB_TOKEN!,
      owner: "batuhanozkose",
      repo: "OzRouter",
    });

    export const GET = configureNextjsAdapter({
      releases: provider.releases,
      assets: provider.assets,
      channels: ["alpha", "beta", "stable"],
    });
    ```

    [electron-releases](https://github.com/lott-ai/electron-releases)

34. **Reference project** â€” `nguyendkn/nextjs-electron` is a production-ready Next.js 16+ Electron template with auto-updater already wired. Uses same stack as OzRouter (App Router, Tailwind CSS 4, electron-builder). [nguyendkn/nextjs-electron](https://github.com/nguyendkn/nextjs-electron)

2. **GitHub Actions workflow** â€” Not researched in detail. Need matrix build YAML for win/mac/linux with code signing secrets
